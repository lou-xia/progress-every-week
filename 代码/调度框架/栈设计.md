## 数据结构

**栈的定义**：

```rust
pub struct StackWapper {
    /// 栈基址
    pub base: usize,
}
```
基本的操作包括: 
1. `new`：调用alloc接口，分配一个栈
2. `dealloc`：调用dealloc接口，回收一个栈
3. 与usize互相转换：通过`From trait`与usize互相转换

**栈池**：

```rust
pub struct StackHandler {
    /// 空栈的集合，这里的Vec是Heapless::vec::Vec
    pub free_stacks: Vec<StackWapper, STACK_POOL_SIZE>,
    /// 当前使用的栈，实现为perCPU的数据。当对应的cpu上运行的栈属于该栈池时，设置为Some(stack: StackWapper)，否则是None
    pub current_stack: [Option<StackWapper>; CPU_NUM],
}
```

基本的操作包括：
1. `new`：初始化栈池，分配出STACK_POLL_SIZE的的栈
2. `alloc_stack`：从`free_stacks`取出栈，没有空闲栈时分配一个新栈
3. `dealloc_stack`：向`free_stacks`回收栈，超过最大数量时直接回收栈
4. `set_current_stack`：设置当前cpu的`current_stack`

**用户态栈池**：

定义为每个地址空间私有数据，每个地址空间都有一个栈池，用来管理这个地址空间的栈的分配与回收。
```rust
pub(crate) static STACK_HANDLER: LazyInit<SpinMutex<StackHandler>> = LazyInit::new();

```
需要考虑加锁

**内核态栈池**：

定义为各个地址空间共享的数据，当各个地址空间或启用双页表，内核使用栈时，从中取出与回收栈。
```rust
vvar_data! {
    KERNEL_STACKS: SpinMutex<StackHandler>,
}
```
也需要考虑加锁

## 栈切换时机

### 同特权级切换

**运行协程**：对应黄色框部分。运行协程时需要保证其使用的栈为空栈。如果当前调度器使用的栈非空，需要从`free_stacks`中取出一个空栈，同时修改`current_task`，修改sp寄存器为这个空栈的栈顶后运行协程。

**运行线程**：对应蓝色框部分。运行线程是需要考虑当前调度器使用的栈是否为空。如果当前调度器使用的栈为空，需要把当前的栈回收到`free_stacks`里，同时设置`current_task`为线程的栈，之后恢复线程上下文，包括修改sp寄存器。这个过程需要提前知道线程使用的栈基址。

这两个过程中，如果调度器使用的是非空栈，不需要额外处理，因为对应的线程已经在tcb里保存了栈底（应该是新的接口）和栈顶（在context里）

此外，同特权级的trap与下面的用户态->内核态的情况类似。

### 用户态->内核态

需要同时维护两个栈池。

**用户栈**：

如果是trap的过程，用户态的栈池没有机会进行保存与恢复，因此不做改变。当第一次从内核态返回这个地址空间的用户态时（即运行的用户态程序不一定是被trap的程序），参考下面的内核态->用户态

如果是在schedule里切换到内核态，用户态栈池可以主动设置`curret_stack[cpu_id]`为None，表示这个地址空间的用户态没有栈在使用（等同于没有任务在执行）。

**内核栈**：

在trap的情况时，trap_entry函数中除了构造trap_ctx外还需要预分配一个新的空栈到sscratch中，保证之后的中断可以直接使用sscratch，这个新的空栈不需要以StackWrapper类型构造，只需要能够转换成usize即可。

> 此外，我们忘讨论什么时候设置内核栈的`curret_stack`了。我的想法是，在参数分析过程中或其前面，这样才能在后面分析的两种结果的跳转都能正常读到`curret_stack`。

### 内核态->用户态

唯一出现的情况为krun_utask

如果utask为协程，需要对内核当前使用的栈回收，即修改`curret_stack`为`None`，把之前的值放回`free_tasks`（超数量就dealloc）（蓝色框部分，但与run_task的区别是设置为None）。

然后从用户栈池中分配一个空栈，设置用户栈池的`curret_stack`为这个栈，然后当作参数传入`into_user`用于回到用户态。修改sp由`into_user`完成。

> 讨论后认为，可以使用非空栈回到用户态，因此我的想法是先判断一下用户栈池的`curret_stack[cpu_id]`是不是None，再决定是否分配栈。

如果utask为线程，蓝色框部分与上述相同。然后从线程上下文中得到线程栈的栈基址，用其设置用户态栈池的`curret_stack`，可以和修改任务状态同时进行。然后运行`into_user_context`回到用户态。修改sp由`into_user_context`完成。
