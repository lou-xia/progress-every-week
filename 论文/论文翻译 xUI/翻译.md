# 扩展的用户态中断(xUI)：无需轮询的快速灵活的通知

## 摘要

扩展的用户态中断(xUI)是一组建立在 Intel UIPI 用户态中断模型上的处理器扩展，以增强其性能和灵活性。本文通过分析和测量，来结构 Intel 当前的 UIPI 设计，并以此开发了一个精确的时序模型。然后，本文介绍了用户态中断的四种新颖增强：跟踪中断、硬件安全点、内核旁路定时器和中断转发。xUI 在 gem5 模拟器中建模，并在三种用例中评估——在高性能用户级运行时中的抢占、使用 DPDK 的 层三路由器的 IO 通知，以及具有流加速器的合成工作负载中的 IO 通知，该加速器模仿英特尔的数据流加速器。这项工作表明，xUI 提供了共享内存轮询的性能，同时具有异步通知的效率。

关键词：中断，用户空间，抢占，处理器架构

## 1. 介绍

随着语言级并发和内核旁路接口的兴起，用户级运行时越来越多地负责曾经属于操作系统的任务——例如抢占、同步和调度（CPU 和 IO）。

用户级线程、goroutine（GO）和协程（C++）使得数据中心应用能够扩展到数百万并发请求 [17, 60]，并且通过避免内核线程的高开销来支持微秒级时间尺度 [26, 37, 48, 52]。这使得 CPU 调度的责任转移到用户级运行时越来越多。类似地，诸如 DPDK [1] 和 SPDK [9] 等通过内核旁路接口通过绕过内核，将 IO 调度转移到用户级别，从而以微秒级的时间尺度实现高性能 I/O。

然而，用户级运行时缺少操作系统赖以协调这些活动的关键支持——中断系统。中断系统提供来自设备、核（IPI）和计时器的低延迟异步通知。异步通知最小化了延迟，因为它立即改变了接收核心的控制流，同时能最大化效率，因为它不需要接收方浪费周期来轮询通知。

如果没有这种支持，用户级运行时将不得不依赖昂贵的操作系统接口，或者退回到以低效率换取更低延迟和更高吞吐量的共享内存轮询 [29]。这直接转化为用户级别的低效率和功能限制。

例如，抢占对于实现对调度的精确控制至关重要。在服务器工作负载中，它可以限制尾部延迟，并通过减轻队头阻塞 [37, 61] 来提高有效吞吐量，并且对于在多租户环境中（如无服务器边缘平台）运行不受信任的代码是强制性的 [21, 59]。然而，许多编程语言和高性能运行时不支持抢占[48, 52]，限制抢占频率（例如，Go 每 10 毫秒抢占一次），或者由于当前用户级定时和通知机制带来的高且不可预测的开销，将不会抢占循环 [6, 7, 10]。

之后，虽然内核旁路接口越来越受欢迎，用于访问高性能网络和存储（例如，超低延迟（ULL）闪存和 NVMe），但接收这些设备提供的微秒级通知的唯一方法是轮询。这会浪费大量的周期和能量，因为核心会忙碌地自旋 [29]。它的扩展性也很差，因为每个被轮询的队列都会占用额外的缓存行，而且处理器没有办法在超过单个队列上空闲（例如，mwait）。专业加速器 [63]（例如，DSA [42]，QAT [33]）的日益普及只会使这一问题更具挑战性。

低成本通知在其他用例中也能实现更高效的实现，包括IPC通知、共享数据结构的同步更改以及处理实时跟踪事件。

英特尔最近发布的用户级处理器间中断（UIPI）为将中断支持引入用户空间提供了一个有希望的起点。例如，它为实现具有低且可预测开销的抢占提供了显著优势[25, 35, 44]。然而，我们对其设计和实现的深入分析（§3）以及对现有用户级通知机制的探索（§2）揭示了其在性能、精度以及对各种通知类型（即定时器和设备）的支持方面的显著局限性。

为了克服这些限制，我们提出了 xUI，这是一组处理器扩展，它建立在 UIPI 之上，以满足用户级运行时的需求。xUI 提供了四个关键功能：跟踪中断（§4.2）——相对于 UIPI，将中断传递开销降低 3 倍至 9 倍，从而缩小了中断和基于内存的通知（轮询）之间的性能差距；硬件安全点（§4.4）——以接近零的成本精确控制中断的传递位置，从而调和了中断和精确垃圾回收之间的矛盾；内核旁路定时器（§4.3）——支持低成本的用户空间定时器，无需昂贵的基于操作系统的定时器，或将内核专用于轮询高精度计数器；中断转发（§4.5）——扩展了中断路由，允许设备将中断发送到用户级线程。

我们在 gem5 中模拟了xUI（和 UIPI）（§5）；并在几个高性能工作负载中评估了其在抢占和 I/O 通知方面的性能（§6），包括：使用 RocksDB 在 Aspen [30] 上提供数据库请求——一个高性能的用户级运行时；使用l3fwd [19] 进行路由——一个基于 DPDK 的第 3 层路由器；以及使用基于英特尔 DSA [42] 的模拟流加速器进行卸载。

我们发现，在 Aspen 中使用 xUI 代替 UIPI 进行抢占，可以将 RocksDB 的吞吐量提高 10%（§6.2.1），并且无需浪费核心来提供高精度计时器（§6.1）。我们发现，在 l3fwd 中使用 xUI 进行 IO 通知，可以达到与忙等待相同的响应速度（延迟），同时显著提高效率。例如，在 40% 负载下，与忙等待相比，xUI 可将 CPU 消耗降低 45%（§6.2.2）。我们观察到，使用 xUI 进行模拟 DSA 的 IO 通知也能获得类似的效率提升；例如，在 50K IPOS（平均请求延迟 20μs）下，xUI 保持与忙等待相同的响应速度，同时产生的 CPU 开销可忽略不计（§6.2.3）。

## 2. 用户级通知的局限性

目前，用户级异步通知和定时的选项存在显著的局限性，特别是对于高性能应用而言。

__信号：高开销，不精确__。在当今大多数系统的用户级别上，信号是异步通知的唯一选择。与中断类似，当没有事件发生时，信号是免费的；但当事件发生时，其开销比轮询高出几个数量级，这使得它们不适合高性能应用程序。

例如，我们测量了信号传递对一些常见基准测试（例如，linpack2、base64 编码）的影响，发现每个信号在大约 2 GHz 时造成大约 2.4 微秒的开销（不含 KPTI）。其中大约 1.4 微秒来自操作系统上下文切换开销。剩余部分主要归因于与内核信号处理代码竞争导致的额外分支预测错误和缓存未命中。

由于信号传递会对吞吐量产生显著影响，因此在高吞吐量应用中，当请求以微秒级的时间尺度到达时 [29, 30, 34]，这种开销使得信号在抢占和 IO 通知方面变得不切实际。即使在常见的服务器工作负载中，请求通常在 100 微秒或更短的时间内 [53, 56]，它们仍然是明显的开销来源。

信号也是不精确的，也就是说，它们会在接收者代码的任意点改变控制流。对于依赖精确垃圾回收的运行时（例如，大多数 Java 和 .NET 实现）来说，这是一个问题，因为精确 GC 依赖于栈图。栈图是在编译时生成的元数据，它告诉 GC 哪些栈元素是指针——每个图仅对程序代码中的特定安全点 [45] 有效。因此，如果一个线程在安全点之外的某个地方被抢占，GC 无法对其进行垃圾回收，因为它无法识别该线程可以引用哪些对象。

近年来，Go 通过从精确 GC 转向混合方法，开发了一种独特的解决方案来解决这个问题 [10]，该方法对最新的堆栈帧使用保守 GC，而对其余部分使用精确 GC。不幸的是，对于大多数运行时来说，这并不是一个解决方案，因为移动垃圾收集器（非常流行）会修改堆栈上的指针 [36]。因此，它们无法容忍保守 GC 的不精确性。

__轮询：不可预测、效率低下、不可扩展__。用户级通知也可以通过轮询共享内存中的变量来实现。当轮询时，每次否定检查都非常廉价（在 L1 缓存命中情况下加载，以及正确预测的分支），并且每次通知仍然相对廉价（在 L2 缓存未命中情况下加载，以及可能错误预测的分支），因此轮询可以支持低延迟和高带宽通知。轮询也是精确的，因为程序员或编译器可以完全控制轮询发生的位置。然而，虽然信号成本随着到达的通知数量而增加，但轮询成本随着执行的检查数量而增加，并且当没有挂起的通知时，很难避免浪费大量周期。

例如，轮询是在编程语言中实现抢占的一种常见方式。为了实现这一点，编译器会添加检测代码，在每个函数入口点和循环回边处轮询一个共享变量，从而确保线程会屈服，而不管控制流如何。不幸的是，轮询开销可能非常高，并且依赖于工作负载 [30]，因为即使是一个简单的检查在紧密循环中也可能非常昂贵。

为了避免这种情况，Go 在最初的十年里没有抢占循环。相反，它要求开发者在必要时在循环中插入对 `runtime.Gosched()` （调度器） 的显式调用。然而，这是脆弱的，当被忽略时，会导致难以调试的减速和程序冻结 [10]。当 Go 团队探索通过向循环添加检查来缓解这个问题时，他们发现这会使程序的几何平均速度降低大约 7%，在最坏的情况下，降低高达 96% [7]。

类似地，我们测量了 Wasmtime 中抢占的成本。Wasmtime 是一个 WebAssembly 运行时，被 Fastly、Shopify、Microsoft 等公司用于运行无服务器应用程序，它依赖于轮询来实现抢占。在一些包含紧密循环或短函数的常见基准测试中（例如，linpack2 [18]），我们发现它导致了高达 50% 的减速。毫不奇怪，当 Java 近年来决定增加对虚拟线程的支持时 [6]，他们决定不支持循环抢占。

轮询 I/O 带来其自身的一系列挑战。正如 Golestani 等人观察到的那样，“你不能总是靠自旋来取胜” [29]。忙于自旋往往会浪费大量的周期，因为 I/O 是不可预测的，并且当 I/O 事件没有到达时，核心就被浪费了。轮询的可扩展性也很差，因为开销随着被轮询队列的数量线性增加，并且每个队列都会消耗一个缓存行。浪费周期会导致能源效率低下，而处理器提供的用于缓解此成本的主要工具（mwait 及其等效项）仅适用于单个队列。降低轮询频率可以减少这些成本，但这迫使人们在开销和响应能力之间进行权衡。在用户级运行时中，定期轮询也很麻烦，因为它与调度纠缠在一起，即，当用户线程正在运行时，无法进行轮询。

__定时器：昂贵且复杂__。定时器支持许多调度任务，包括抢占、周期性设备轮询和同步超时。不幸的是，用户级运行时对于微秒级定时的选择有限。硬件定时器隐藏在昂贵的操作系统接口之后，这些接口依赖于信号 (`setitimer()`, `timer_create()`) 和系统调用 (`nanosleep()`, `select()`) 进行通知。

在缺乏直接访问高效硬件计时器的情况下，实现线程的用户级运行时 [7, 30, 44, 54] 通常指定一个内核线程（核心）作为计时器，并可以通过信号、共享内存以及最近的 UIPI，在其他内核线程需要被抢占时通知它们。不幸的是，对于计时器线程，操作系统开销会很快累积（参见图 6）。在更高性能的系统 [30, 34, 44] 中，通过在高精度计时器（rdtsc）上忙等待来实现计时器线程是很常见的，这会消耗整个核心的资源。

__UIPI：比轮询更昂贵、不精确、范围窄__。使用 UIPI 接收通知大约比信号便宜 3 到 5 倍——大约 2800 个周期，而在我们的 Saphire Rapids 处理器上是 600-900 个周期。不幸的是，UIPI 仍然比基于内存的通知慢大约 6 到 9 倍，而基于内存的通知大约是 100 个周期。

由于推测窗口的尺寸不断增大，未来处理器中 UIPI 和轮询开销之间的差距将会扩大。即，由 UIPI 引起的流水线刷新是开销的重要来源（§3.5）。随着未来处理器中在途指令数量的持续增加，这将变得更加昂贵。这种趋势也会损害未来的轮询性能，因为处理分支预测错误的刷新会变得更加昂贵。

UIPI 同样不够精确，类似于信号，因此它与精确式垃圾回收机制（GC）不兼容。最后，UIPI 仅支持 APIC 提供的一小部分功能，即不支持设备中断或定时器中断，这限制了它的实用性。总而言之，这使得 UIPI 在某些情况下成为一个有用的工具 [25, 30, 35, 44]，但仍然远不能替代中断系统所提供的所有功能，当然也不是高性能系统中轮询的可行替代方案。

## 3 Intel UIPI 的特性

在此，我们通过逆向工程其微架构细节，首次对英特尔 UIPI 进行了深入分析。我们将在第 4 节讨论 xUI 如何在此基础上构建。

### 3.1 UIPI 概述

用户处理器间中断（UIPI）支持是英特尔最近在其 Sapphire Rapids 处理器中引入的。UIPI 允许 IPI 直接从一个内核线程（例如 pthread）发送到另一个用户态内核线程，从而避免了信号的上下文切换开销。

__增加访问控制__。访问控制对于防止线程向其他未经同意的线程发送 IPI 是必要的。UIPI 通过一个新的硬件接口（通过 MSR 和内存表编程）将此任务委托给内核，该接口称为用户中断目标表（UITT）。UITT 是一个按进程的表，它指定一个进程（及其所有线程）可以向哪些其他线程发送UIPI。从高层次来看，每个条目都是一个元组
⟨`UPID`, `vector`⟩——其中 UPID（用户发布中断描述符）是一个线程特定的内存描述符。在一个进程的 UITT 中储存 UPID 指针，这隐式地授予了该进程向与该 UPID 关联的线程发送中断的权限。内核允许授权进程使用系统调用来建立这些映射。正如我们稍后讨论的那样，UITT 及其 UPID 条目对于 UIPI 路由也是必不可少的。

__用户级路由中断__。中断系统负责将消息从源（设备、核心、定时器）路由到目的地。目的地是核心（通过 APICID 寻址），消息是8位值（向量）唯一地标识中断的原因。按照设计，路由是静态的——APICID 通常在启动时分配给核心，并且很少更改。

为了使用户中断扩展 x84-64，需要克服现有中断路由方案的几个限制：（1）没有寻址线程的概念——线程是动态创建、销毁并在核心之间迁移的。（2）x86 每个核心（APICID）的向量空间非常小（256个可能的值），并且其中一部分已经专用于其他目的——因此与用户 IPI 共享这个空间将非常有限。（3）核心始终可用，但线程并非如此；当一个中断到达时，目标线程可能已被上下文切换出去。

为了解决这些局限性，UIPI 创建了自己的正交虚拟命名空间（线程 ID）和向量空间（一个6位用户向量，或 UV）。为了在这个新的命名空间中传递中断，它引入了两种新的数据结构，这些数据结构在核心之间共享内存——即之前提到的每个进程的 UITT 和每个线程的 UPID。接下来，我们将更详细地了解这些数据结构，以及 UIPI 如何使用它们来发送和接收中断。

### 3.2 发送和接收 UIPI

从高层次来看，当发送一个 UIPI 时，发送核心会在 UITT 中查找目标线程的 UPID，修改目标线程的 UPID，然后向接收核心发送一个传统的 IPI，以通知它有一个待处理的 UIPI。然后，接收核心使用 UPID 中的状态来传递请求。

在此基础上，我们首先注意到，每个可以接收 UIPI 的线程都有一个由内核分配的关联 UPID。每个 UPID 都是一个在所有核心之间共享的内存中的结构体。我们将在下面的描述中引用表 1 中显示的其字段。

> 表1. 用户发布的 Interrupt Descriptor（UPID）字段
> | 名称 | 说明 | 位范围 |
> | - | - | - |
> | 待处理通知 (ON) | 是否有一个或多个用户态中断 (UI) 有待处理的通知 | 0:0 |
> | 抑制通知 (SN) | 发送方是否应避免发送通知 | 1:1 |
> | 通知向量 (NV) | UI 应发送中断到哪个 CPU 的向量 | 23:16 |
> | 通知目标 (NDST) | 此线程运行所在核心的 APICID | 63:32 |
> | 已发布的中断请求 (PIR) | 已发布到此线程的中断 | 127:64 |

首先，在发送 UIPI 之前，用户空间程序会请求内核设置路由[5]。在接收线程上，`register_handler(. . .)` 系统调用会分配一个 UPID，并存储一个指向用户级中断处理程序的指针，该处理程序将在收到 UIPI 时被调用。在发送进程上，`register_sender(. . .)` 系统调用会在 UITT 中分配一个新条目，并在那里存储一个指向相应 UPID 和用户提供的向量的指针。

用户中断通过新的 `senduipi` 指令发送，该指令接受一个操作数，即 UITT 中的索引。要发送 UIPI，发送核心在 UITT 中查找 UPID 和向量，设置 UPID 的 PIR 字段中的位以指示已发布的向量，并设置 ON 位以指示存在待处理的中断。然后，它向运行线程的核心发送一个传统的 IPI（NDST 中的地址），向量（NV）指示存在待处理的 UIPI。

当此 IPI 到达时，接收核心会将接收到的向量与 MSR（UINV）的字段进行比较，该字段存储 UIPI 通知的向量。如果它们匹配，则接收器会查询当前线程的 UPID 的 PIR 字段，以确定是否应将 UIPI 传递给当前线程，或者是否需要由操作系统处理。

如果 UIPI 注定要发送给当前运行的线程（快速路径），它会通过调用接收者注册的处理程序来传递它，并在完成后使用新的 uiret 指令返回。否则，它将在稍后通过内核在慢速路径上进行传递。在慢速路径上，当内核恢复中断所指向的线程时，它将通过本地 APIC 将捕获的 UIPI 重新发布为自 UIPI。

然而，如果我们频繁接收和处理那些目标为非上下文进程的中断（慢速路径），这可能会降低在接收核心上运行的其他程序的速度。为了缓解这个问题，UIPI 在 UPID 中包含一个“抑制通知”（SN）位字段，指示接收器何时未准备好，从而阻止发送者继续向该线程发送中断。当一个线程被上下文切换出去时，内核会设置 SN 位，从而阻止来自发送者线程的进一步 UIPI。

线程也可以使用新的 clui 和 stui（分别清除和设置用户中断标志）手动切换中断传递，它们的工作方式与内核模式下的对应指令类似，并使用 testui 查询中断是否被阻止。

最后，接收线程可能会从一个物理核心迁移到另一个物理核心。因此，发送核心必须正确确定它应该将 IPI 发送到哪个目标核心。为了解决这个问题，发送核心读取 UPID 的 NDST 字段，该字段存储线程当前运行的核心的 APICID。因此，要将线程迁移到不同的核心，操作系统只需更新此字段。

### 3.5 解构 UIPI 微架构

__中断处理策略__。处理器处理中断时采取的第一步是将控制流重定向到中断处理程序。然而，由于现代处理器中的乱序执行缓冲区能够容纳 500 条或更多在途指令，因此它们在处理这些在途指令时所选择的策略对性能有重大影响。_刷新_ 所有在途程序指令是传统的解决方案，因为它最大限度地减少了处理中断的延迟。但是，_排空_ 所有在途指令也是一种选择。在这里，所有（正确获取的）在途指令都将被允许完成。一旦流水线中没有中断前的指令，程序计数器将指向中断前程序中的最后一条指令。可以保存下一条指令的 PC，指定处理器在中断处理程序之后应从何处恢复执行。这种方法引入了更大的延迟（等待流水线清空），但最大限度地减少了浪费的工作。

为了发现我们的 Sapphire Rapids 处理器的功能，我们构建了两个测试程序来揭示此特性的效果。我们利用了这样一个事实：通过刷新，UIPI 的端到端延迟与接收核心上的在途指令无关。采用排空策略，延迟取决于在途指令的类型。

我们的第一个测试程序执行指针追踪，允许我们改变解析/排空链所需的时间。我们逐渐增加工作负载的大小，增加缓存未命中指针的百分比，从而增加从流水线中排空相关获取指令链所需的时间。我们观察端到端 UIPI 延迟的变化，预期随着缓存未命中的增加，如果采用排空策略，这将影响端到端 UIPI 延迟。然而，我们没有观察到这种延迟变化，表明我们的处理器采用了刷新策略。我们的下一个测试程序使用英特尔的硬件性能计数器来测量当接收到中断时，已分派但未提交（即，刷新）的指令数量。我们发现刷新的微操作数量与接收到的中断数量呈精确的线性增长关系，再次表明采用了刷新策略。

__接收者开销来自哪里__？为了理解接收用户中断的成本，我们设计了一个测量程序，该程序记录了接收 UIPI 的每个单独步骤的成本，如 §3.3 中所述。由于每个步骤都会对共享数据结构（例如，UPID）产生可观察的影响，因此我们并行线程中运行的测试程序可以监视这些位置，以查找接收核心所做的更改，并测量这些事件发生之间的时间。由于这些更改发生在内核模式下，因此我们的测量程序被实现为一个内核模块。

如果我们的测量程序和观察到的接收线程在不同的物理核心上运行，那么对 UPID 的内存写入必须遍历最后一级缓存——从而给我们的测量引入不可预测的开销。为了避免这种情况，我们将我们的测量内核模块绑定到接收线程的同级 SMT 上下文中。

为了确定中断何时开始，我们需要记录在处理中断之前要执行的正常程序流程中最后一条指令的时间戳。为了便于此操作，接收进程在一个循环内执行 rdtsc 指令以记录当前时间。在中断处理程序中，此记录的值将保存为中断前执行的最终点。

## 4 xUI：扩展用户中断

xUI 尝试满足以下设计目标。

- __快速__：越快越好；通知是 IO、抢占、同步等的基本构建块。xUI 旨在提供与共享内存通知相当的性能，以消除轮询的需要。
- __非侵入性__：xUI 的采用应尽可能减少对现有高性能处理器的改动。因此，它应该具有较小的材料清单，需要对现有 ISA 和微架构进行最小的更改，并且不会对其他功能造成性能损失。
- __简单且兼容__：xUI 应与现有软件协同工作，并实现比当前用户级别通知更简单的解决方案。
- __通用性__：xUI 应支持中断系统提供的所有形式的通知，包括定时器、设备中断和 IPI。

### 4.1 概述

__跟踪中断：弥合内存差距__。跟踪中断提供了一种替代中断传递的方法，避免了刷新流水线的高昂成本。这使得通知的延迟和性能成本与基于内存的通知（轮询）相当，同时具有异步通知的效率和响应性。跟踪中断将 UIPI 的开销大约降低了 3 到 9 倍，根据中断类型（IPI、设备、定时器中断），通知分别需要 231 或 105 个周期。这为高性能应用中的性能和效率提供了显著的优势（§6）。

__硬件安全点__：用于精确 GC 的精确中断。硬件安全点提供了一个显式的安全点指令。这允许编译器精确地传达在何处传递中断是安全的。安全点以接近零的成本满足了具有精确 GC 的语言的需求，并且架构复杂性极低。虽然 UIPI 可以使用 clui 和 stui 来实现这一点，但这对指令总共花费34个周期（表 2），这已经足够昂贵，以至于当在紧密循环或热代码路径中使用时，它们的成本高得令人望而却步。例如，在 RocksDB 的 `malloc()` 中使用它们来保护关键部分会导致 7% 的吞吐量损失。

__KB_Timer__：快速高效的用户级定时器中断。KB_Timer（内核旁路定时器）消除了基于操作系统的定时器和专用定时器线程的成本和复杂性。

使用 KB_Timer，每次定时器中断为 105 个周期。由于每个核心都有自己的定时器，因此消除了随核心数量增加而增加的额外通知开销（图 6）。KB_Timers 也可以直接从用户空间进行编程，从而可以高效地访问以更改定时器频率、禁用定时器或使用不同的操作模式（单次与周期性）。

__中断转发__：线程的设备中断。NIC 和加速器的内核旁路接口强制用户级代码轮询 IO 完成事件。xUI 旨在缓解这种低效率。设备中断通常被路由到核心（通过 APICID）而不是线程，并且 UIPI 不提供解决方案。中断转发（§4.5）通过将特定 APICID/向量的中断路由到用户级线程/向量来克服此限制，从而允许用户级代码通过设备享受 xUI 的优势。

### 4.2 跟踪中断

目前 CPU 流水线中的中断处理策略需要在艰难的权衡中做出选择，要么牺牲指令吞吐量，要么导致高延迟。跟踪中断为应对推测执行的挑战（§3.5）提供了一种新策略，该策略可在不牺牲吞吐量的情况下最大限度地减少中断处理延迟。

概括而言，_刷新_——现代乱序处理器上的常用方法——在接收到中断时立即刷新流水线。这会丢弃所有正在进行的工作，从而损害吞吐量，但可以最大限度地减少延迟。在具有超过 500 条在途指令的现代架构中，每次中断时（对于被中断的线程）损失的吞吐量可能代表数百或数千个周期的进度损失。也可能存在延迟成本，因为许多架构会限制在一个周期内可以取消的微操作的数量。另一种可能的方法是 _排空_——在传递中断之前，先完成所有在途指令。这不会浪费工作，但会增加显著的延迟。这种延迟是必要的，因为在解决所有推测来源之前，我们无法开始获取指令；否则，我们可能会丢失中断。这两种方法的成本都在增加，因为连续几代的处理器都在增加指令窗口的大小——从而能够进行更多在途工作。

我们的方法，即 _跟踪_，通过利用中断本质上是异步的这一观察结果，避免了这两种妥协。因此，处理器在指令流中发出中断处理代码的位置具有一定的灵活性。具体而言——刷新和排空都将检测到中断信号的指令流中的精确点视为有意义的，即，作为一个同步事件，并在该指令之后立即发出中断处理微操作（分别是最后一条已完成的指令或最后一条已获取的指令）。

跟踪利用了这样一个事实：指令流中检测到中断的精确点在语义上没有意义——中断本质上是异步的，因此处理器可以选择在何处发出中断处理程序，以在不牺牲吞吐量的情况下实现最佳可能的延迟。

尽管跟踪会将指令附加到在途指令流的末尾，比如排空，但我们并没有遇到相同的延迟。因为我们（立即）注入指令，而这些指令不依赖于先前的指令，所以我们预计它们会迅速进入执行阶段，且延迟成本极低，即使在存在非空指令窗口的情况下也是如此。

__微架构设计__。与刷新或排空不同，跟踪主要在处理器的前端（取指单元）实现，而不是在后端（提交单元）实现。

从高层次来看，跟踪分两步传递中断。首先，一旦本地 APIC 接收并接受中断，跟踪会强制取指单元将控制流重定向到 MSROM，其中存储了用于中断通知处理的微操作——有效地将这些指令注入到微操作流中。接下来，它会跟踪这些微操作是否已提交，以确保：（1）如果处理器由于错误预测而刷新较新的在途指令，它们将被重新发布；（2）我们可以准确地识别并保存最后一条程序指令——执行将在中断处理程序运行后恢复的位置，即在（iret）指令之后。这种方法保证了中断最终将被传递，即使在途分支被错误推测，也不会丢弃有用的工作。图 3 概述了我们方法引入的架构变更。

现在让我们更详细地检查跟踪。首先，当本地 APIC 发出中断时，前端的“下一 PC 逻辑”会检查它是否位于指令边界——确保它不是在解码单个指令的微操作的过程中。一旦满足此条件，它会立即将微程序计数器（micro-PC）设置为 MSROM 中中断通知处理微例程的开头，并继续执行。

接下来，由于我们既不刷新也不排空，因此我们需要应对几个新的挑战：

首先，我们必须处理错误预测的可能性。例如，在分支预测错误时，处理器通常会通过刷新所有比分支指令更年轻的指令来恢复执行，然后继续执行。然而，这也将刷新我们的中断处理代码，导致我们丢失一个中断。

为了防止这种情况，我们在前端实现了一个状态机，以跟踪我们是否正在处理中断。如果在处理中断时由于错误预测而发生刷新，我们会立即将取指单元重定向到我们的中断处理微代码——在恢复执行之前重新注入这些微操作。

在首个中断微操作提交之前，中断处理微代码将保持为默认的错误预测恢复路径。处理器有可能发生错误预测并多次重启，但是，中断处理微代码——以及中断处理程序，最终将会被提交。后续的重启可能会更快，因为中断处理程序的代码/数据可能已经在缓存中。

其次，我们需要能够识别中断发生前最后一条已提交程序指令之后的 next_pc 值，以便在执行 iret（从中断处理程序返回）时返回到正确的地址。这个正确的返回地址可能会随着我们处理错误预测而改变。然而，我们解决错误预测的方案也能无缝地处理这个问题，而无需显著改变中断处理微代码。首先，当重定向执行时，我们从前端读取 next_pc。如果我们没有观察到错误预测，那么这个值是正确的。如果从错误预测中恢复，前端将（像往常一样）计算一个新的 next_pc 值，我们的重启中断处理微代码将再次使用它。

__比共享内存通知更便宜吗__？我们的结果表明，对于接收方而言，跟踪中断可能比共享内存通知更便宜，原因与它们比普通中断更便宜的原因类似——共享内存通知本质上是同步的，因此它可能会引入意外的控制流。

即——轮询通常涉及读取共享变量，常见的情况是缓存命中且没有待处理的通知。当通知到达时，它通常涉及一个长时间的内存操作（由于远程写入导致的失效事件引起的缓存未命中，随后是对远程行的慢速读取）和一个分支预测错误——导致刷新比错误预测的分支更晚的指令。此外，这些步骤将被序列化，因为只有当长延迟加载检索到其值时，分支预测错误才会解决，恢复才会开始。同样，与中断类似，随着未来处理器中推测窗口的增长，这种成本将会增加。

目前，跟踪中断是否像轮询一样廉价（或更廉价）取决于中断路由的成本。回顾一下 §3.3 中描述的传递 UIPI 的步骤。在没有刷新成本的情况下，UIPI 的下一个主要开销是读取 UPID，这相当于轮询，因为如果远程核心已更新UPID以发送IPI，则同样会错过缓存。但是，正如我们接下来讨论的那样，使用 xUI 的定时器（§4.3）和设备（§4.5）中断都不需要修改此结构。因此，用于中断传递的微代码可以从 §3.3 中描述的步骤 5 开始，该步骤仅需执行 105 个周期。

__硬件成本和复杂性__。我们实现跟踪中断的方法旨在最大限度地减少现有高性能 CPU 流水线中的额外成本和复杂性。例如，当前的处理器不支持部分刷新，因此 xUI 不依赖于它们。因此，如果在检测到中断之前获取的指令发生错误推测，则正常的错误推测恢复逻辑会刷新错误推测之后的所有指令，包括来自中断路径的指令，即使我们稍后需要重新发布这些指令。虽然部分刷新可能潜在地提供一些性能优势，但改进似乎微不足道，不值得增加复杂性 [15, 20]。

我们的材料清单包括：每个 ROB 条目一位，用于指示微操作的来源是来自正常的程序执行还是中断路径（即，中断处理微代码或中断处理程序）；以及每个从前端到后端的微操作一位，用于将此来源信息传递到 ROB；少量逻辑门，用于实现下一 PC 逻辑，以便在中断时将指令提取引导至中断通知微代码（图 3）；少量逻辑门和一根导线，用于扩展提交逻辑以检测是否已提交中断并向前端发出信号，以便前端知道无需再次发出中断。

### 4.3 内核旁路（KB）定时器

内核旁路定时器（KB_Timer）提供了一个每线程的用户级定时器，类似于本地 APIC 定时器。与 APIC 定时器一样，它不适合应用程序开发者直接使用。相反，它提供了一个底层原语，用户级运行时可以使用它来实现软件定时器，用于诸如抢占、周期性轮询、超时等任务。

定时器通过两个新的指令使用：`set_timer(cycles, mode)` 和 `clear_timer()`。`cycles` 是一个 64 位无符号整数，`mode` 是一个一位标志，指示定时器是周期性的还是单次的。如果 `mode` 是周期性的，cycles 被解释为周期，单位是 cycles。如果 `mode` 是单次的，cycles 被解释为一个截止时间，同样，这与传统的 APIC 设计保持一致，使得在实现多个软件定时器时可以简单地指定下一个截止时间。

__微架构设计__。每个物理核心有一个 KB_Timer，它通过操作系统在多个线程之间进行多路复用（如下所述）。KB_Timer 有其自己的物理定时器，而不是简单地重用本地 APIC 定时器，因为内核已经在使用该定时器。与现代本地 APIC 定时器类似，该定时器使用系统时钟来提供高分辨率。

KB_Timer 中断通过调用用户级中断处理程序来传递，并向其传递内核分配的向量。KB_Timer 中断传递非常经济（105 个周期），因为它能跳过与 UPID 和路由（`notification_process`）（§3.3）相关的微代码步骤，并直接调用 `interrupt_delivery` 微代码。

与 IPI 不同，没有特殊的慢速路径支持，即，它不打算在内核或所有者以外的任何线程正在执行时触发。如果定时器在内核态下达到其目标，它将陷入。如果它处于用户态，它将向当前正在执行的任何线程传递一个中断。因此，由内核来管理定时器状态。

__KB_Timer 的多路复用__。内核通过写入 `kb_config_MSR` 来启用和禁用 KB_Timer，并设置定时器中断向量。与 UIPI 类似，这允许内核通过系统调用（`enable_kb_timer()`, `disable_kb_timer()`）来控制对此机制的访问。

为了保持每个内核线程模型的一致性，在上下文切换时，操作系统通过读取 kb_timer_state_MSR 来保存当前计时器，该 MSR 返回计时器截止时间（时间 + 当前时间戳），并将其与分配的向量、周期和模式一起保存，以便在线程恢复时可以恢复计时器。

在慢速路径上（即，从内核传递中断），我们选择简单地在上下文恢复时检查截止日期是否已超过，并传递中断——类似于正常的 UIPI 慢速路径。然而，内核也可以在线程未运行时，继续使用内核定时器（最终使用本地 APIC 定时器）跟踪定时器，并在定时器到期时立即重新调度线程。

### 4.4 硬件安全点

正如 §2 中所讨论的，由于 UIPI 和信号无法精确控制中断的传递位置，这与需要精确垃圾回收的语言的需求相冲突，因为这些语言期望在安全点被抢占。概括地说，安全点是代码中的精确位置——在编译时通过静态分析确定——在这些位置，精确 GC 所需的预先计算的堆栈映射是有效的[10]。

硬件安全点通过为编译器提供一种精确且近乎零成本的机制来解决这个问题，编译器可以指示处理器在何处安全地传递中断。从概念上讲，安全点可以被认为是一种特殊的 NOP（空操作）——一个安全点指令。它们的实际实现取决于具体的架构。在 x86 上，安全点可以使用 x86 指令前缀[4]进行编码，有效地将任何指令转换为硬件安全点，从而消除了对单独指令的需求。由于安全点是具有精确GC的语言的编译器中的常见原语（例如，LLVM IR中支持它们[45]），因此可以修改编译器以发出安全点，而只需进行最小的更改。xUI 的安全点模式——仅在安全点传递中断——通过系统调用进行切换。

__微架构设计__。为了启用安全点，我们添加了一个一位标志寄存器（实现为类似于 UIPI 的 UIF 标志的 MSR），以指示是否启用了安全点模式。当安全点标志被设置时，处理器仅在安全点指令处传递中断。

我们的跟踪中断（§4.2），与普通的处理器中断一样，只在指令边界传递，确保它们不会在单个指令的微操作执行过程中传递。因此，实现安全点只需要对现有的指令边界检查进行微小的修改。

具体而言，我们通过附加条件扩展了此检查：如果启用了安全点模式，处理器将验证当前正在解码的指令是否为安全点指令，并且仅在此情况下传递中断。预测在此处增加了一些复杂性，因为我们必须处理在错误路径上推测性地获取和解码安全点的情况。

在从错误预测（通常带有跟踪）恢复期间，如果存在待处理的中断，处理器将重定向到中断处理路径，而不是跟踪的程序 PC。但是，如果处理器处于安全点模式，则在恢复后无法立即处理中断，因为安全点位于错误预测的路径上。在这种情况下，处理器进入一种状态，其中正常执行会持续到遇到另一个安全点，然后跟踪的中断逻辑会将处理器重定向到中断处理路径。

当通过优化的前端路径解码安全点指令时，会出现另一个复杂情况，例如英特尔架构中的微操作缓存或循环流检测器。为了确保安全点与这些优化协同工作，我们在每个微操作的编码中添加一位，以指示它是否为安全点。如果处理器处于安全点模式且有中断挂起，并且它正在这些优化路径之一上执行微操作，则它使用此位来检测它是否位于安全点，因此允许传递中断。

### 4.5 中断转发：路由设备中断

我们希望利用 xUI 来实现设备 I/O 的优势，然而，UIPI 有其自身的交付方案，使用中断提供单元标识符（UPID），而中断系统的其余部分对此一无所知。它只知道如何将中断路由到 APICID，这些 APICID 通常在启动时分配给内核。

中断转发通过扩展本地 APIC 来解决这个问题，允许它将原本发往核心的中断直接转发到当前线程。如果目标线程没有运行，本地 APIC 会通知操作系统来处理该通知，类似于 UIPI。为了实现这一点，中断转发允许内核在每个核心（APICID）向量和用户级向量之间建立映射。因此，当一个中断到达一个映射的向量时，该中断会被转发，很像网络中的端口转发。

为了建立映射，线程通过操作系统注册以接收来自设备的中断，并被分配一个通知向量。中断通过中断处理程序正常传递。然而，与 KB_Timer 类似，UPID 永远不会被触及，因此快速中断传递路径（接收线程正在运行）更快，并且没有共享内存的争用。

__微架构设计__。中断转发通过两个新的 256 位寄存器 `forwarding_enabled` 和 `forwarded_active` 扩展了本地 APIC。这些寄存器中的每一位对应于一个向量。`forwarding_enabled` 指示要在当前核心上转发哪些中断。`forwarded_active` 指示应将哪些中断转发到当前正在运行的线程。

例如，如果一个中断到达，其 _向量_ == 8。如果在 `forwarding_enabled` 中设置了第 8 位，则 APIC 将在 UIRR MSR 中设置第 8 位。从这里开始，有两种选择：快速路径和慢速路径。

如果 `forwarding_active` 中的第 8 位被设置，APIC 将通知核心传递一个中断——这是我们的快速路径，中断直接传递到用户态线程。如果第 8 位未被设置，这意味着目标线程当前未运行——这是我们的慢速路径。在这种情况下，APIC 发出一个常规的传统中断，内核陷阱处理程序查看原因，然后从 UIRR 读取向量，并将其存储在 DUPID 中以供稍后传递（见下文）。

__中断转发的多路复用__。每个内核任务（线程）都有其自己的 256 位向量，指示哪些中断被转发给它。当线程恢复执行时，此向量被写入 `forwarded_active`。

在注册期间，操作系统还会分配一个类似于 UPID 的数据结构，用于处理每个接收器上的慢速路径——即当目标线程未运行时，我们称之为设备用户中断发布描述符（DUPID）。当转发的中断到达，而当前线程没有等待该中断时，内核会更新 DUPID，并通过正常的慢速路径传递中断，类似于 UIPI。

我们的中断转发方案虽然高效且侵入性极小，但受到底层核心有限向量空间的限制，因为该空间必须由主机上的线程共享。这种限制约束了可以同时支持的设备/用户对的数量。人们可以想象在中断系统的消息格式中添加一个新字段，或者重新利用现有消息格式中未使用的位（例如 clusterID）来避免这种限制。

## 5 实验方法

### 5.1 硬件平台

我们所有的实验均在一颗配备 28 个 Sapphire Rapids 核心的 Intel Xeon Gold 5420+ 处理器上进行，使用的 Linux 版本为 Intel 分支的 v6.0.0，并带有 UIPI 补丁[5]。我们以 2 GHz 的频率运行处理器，并禁用了 TurboBoost、频率缩放和 c- 状态，以确保可控的测量。我们同时使用了 Linux perf [23]和 Agner 工具[22]来读取性能计数器。为了估计被刷新的微指令数量，我们计算了已提交的微指令数量与已解码的指令数量之间的差值——因为没有直接的性能计数器用于统计被刷新的微指令。

### 5.2 gem5 模拟

我们的模拟基于 gem5（版本23）[46]，并扩展了乱序 CPU 模型。我们配置了基线架构，以模拟英特尔 Sapphire Rapids 处理器，例如英特尔至强金牌 5420+，使用的参数如表 3 所示。我们再次以全系统模式运行 gem5，使用的是英特尔的 v6.0.0 Linux 内核分支[5]。我们在 gem5 中实现了 UIPI 支持，并使用我们的特征研究（§3）来提供准确的刷新、通知和传递成本，并在此基础上构建了对 xUI 的支持。我们还验证了我们在 gem5 中 UIPI 模拟的接收器开销与我们的 Sapphire Rapids 处理器上的 UIPI 开销非常匹配（表2），两者都在 720 纳秒左右。为了支持我们的 I/O 通知实验（§5.4），我们将 gem5-dpdk[58] 设备模型从 ARM 移植到 x86，并添加了使用 xUI 传递 I/O 完成的支持。

一个有趣的发现是，gem5 的中断性能模型与真实硬件截然不同。例如，它会排空流水线而不是刷新它，并且在每次排空后人为地添加了固定的 13 个周期。为了解决这个问题，我们计划将我们改进后的中断模型向上游贡献给 gem5。

### 5.3 采用UIPI和xUI的抢占式调度

为了评估使用抢占式调度的 UIPI 和 xUI，我们在 gem5 中运行 Aspen [30]。Aspen 是一个基于 Caladan 的用户级运行时系统 [26]。它支持轻量级的用户级线程和内核旁路 I/O。Aspen 通过用户中断实现抢占式调度，并使用工作窃取来平衡各个核心之间的线程。

对于该应用，我们运行 RocksDB [8] (v5.15.10)，这是一种流行的键值存储，使用双峰工作负载，包括 99.5% 的 GET 请求（1.2 微秒）和 0.5% 的 SCAN 请求（580 微秒）。我们使用 Caladan 的开环负载生成器生成请求，该生成器配置为使用 UDP 和泊松到达分布。负载生成器和 RocksDB 服务器都在单个核心上的 Aspen 运行时中运行。

为了使我们能够在 gem5 中运行此工作负载，我们通过三种方式调整了 Aspen。首先，为了在 gem5 中模拟网络流量，我们添加了一个额外的核心，用于在负载生成器和 RocksDB 服务器之间转发数据包。其次，我们禁用了核心重新分配以减少噪声，并删除了实现它的内核模块（ksched）。相反，我们将每个内核线程绑定到一个特定的 CPU 核心。最后，我们扩展了 Aspen 以使用我们的 KB_Timer。启用 KB_Timer 后，每个核心都有自己的计时器。

### 5.4 IO 通知：L3Fwd & 模拟加速器

__基于 DPDK 的 L3 转发__。我们采用了与先前工作类似的方法来测量与使用 DPDK 的自旋轮询相关的低效率[29] 。在我们的实验中，我们使用 DPDK 的第三层转发（L3Fwd）应用程序，采用最长前缀匹配（LPM）算法，包含 16,000 个条目的路由表和 64 字节的 IPv4 UDP 数据包。我们对具有 1、2、4 或 8 个不同网卡的系统进行建模，每个网卡都有自己的接收队列。数据包被发送回同一个网卡，在 1 个网卡的情况下，类似于[29]。

我们使用了一个专用的外部设备和一个开环数据包生成器，它们是 gem5-dpdk [58]的一部分，以生成我们的实验工作负载。我们修改了数据包生成器，使其使用指数分布来表示数据包的到达间隔时间，以便更准确地模拟真实网络流量的突发性[14, 27]。

__片上加速器__。我们构建了一个类似于英特尔 DSA 的模拟片上加速器[42]，具有可配置的卸载延迟分布。该加速器通过模拟的 PCIe 总线连接到我们的 gem5 系统。用户程序使用类似于 SPDK 的异步接口将卸载提交到模拟加速器[9]。我们使用一个综合的闭环工作负载评估了我们的加速器，该工作负载在等待请求完成时会进行忙等待。

实际的卸载延迟随请求的操作、请求（缓冲区）大小以及加速器的争用情况而变化。为了捕捉这种可变性，我们通过在加速器的响应时间上增加不同幅度的随机噪声来模拟卸载延迟。我们模拟了两种类型的请求，其响应时间分别为 2 μs 和 20 μs。根据先前的工作 [42] 和我们自己的测量，具有这些平均卸载延迟的操作和数据大小的示例包括：（1）对于 2 μs 的延迟：使用 DSA [42] 复制一个 16 KB 的缓冲区，或者复制一批最多 2048 字节的 8 个缓冲区；（2）对于 20 μs 的延迟：使用 DSA [42] 复制一个 1 MB 的缓冲区。

## 6 评估

### 6.1 xUI 特性

下面我们将评估 xUI 的接收端开销，对比两种精确机制（即硬件安全点和轮询）的接收端开销，考察 KB_Timers 对发送端开销和核心数量的降低作用，并探索 xUI 中断的最大延迟。

__接收端开销__：UIPI 和 xUI。首先，我们比较了 xUI 和 UIPI 在接收中断方面的开销。图 4 展示了周期性中断传递对三个不同基准测试（fib、linpack 和 memops）的影响。

我们的实验会向运行中的应用程序发送周期性中断，并测量完成所需的时间增加了多少。我们研究了三种不同的策略来隔离不同 xUI 机制的优势：一个专用的定时器核心发送 UIPI（UIPI SW Timer），xUI（SW Timer + Tracking）——跟踪的中断也由一个专用的定时器核心发送，以及 xUI（KB_Timer + Tracking）——跟踪的中断由 KB_Timer 发送。

首先，我们可以看到，与 UIPI 基线相比，跟踪中断将传递中断的每个事件成本从平均 645 个周期降低到跟踪下的 231 个周期。将 KB_Timer 作为我们的时间源可以消除访问 UPID 所带来的额外传递成本，如 §4.3 所述，将中断的每个事件成本进一步降低到平均 105 个周期。因此，以 5 μs 的间隔传递中断的成本从 UIPI 的 6.86% 降低到 KB_Timers 和跟踪下的 1.06%，约降低了 6.9 倍。

__硬件安全点与轮询式抢占__。轮询式抢占。我们将线程抢占的开销与两种精确机制——xUI 硬件安全点和轮询，以及 UIPIs 进行了比较。我们的实验抢占了两个程序：matmul 和 base64。为了实现轮询式抢占，我们采用了 Concord [34]，一个最先进的基于编译器的插桩（轮询）抢占系统，作为代表性系统。为了评估硬件安全点，我们修改了 Concord 的编译器插桩通道，以插入安全点指令而不是轮询检查。

图5显示，对于任何抢占量子，硬件安全点比其他两种方法在两个程序上的开销都低得多。在 5𝜇s 的间隔下，我们的硬件安全点仅显示 1.2%-1.5% 的减速——主要归因于上下文切换的开销——而基于编译器的方法则带来 8.5%-11% 的开销。这种改进主要归功于（1）xUI 较低的接收器成本，以及（2）我们的安全点指令的近乎零成本，因为当没有事件时，它们仅被解释为 NOP。请注意，由于我们在处理抢占后返回到同一线程，因此与上下文切换到不同线程、I/O 处理程序等相比，上下文切换的成本被最小化——这将污染微架构状态。虽然这具有隔离由于通知成本带来的改进的优点，但它并没有说明这些好处如何在更复杂的设置中被分摊。我们在 §6.2.1 中对 RocksDB 的端到端评估捕捉了这些更实际、更高的上下文切换成本。我们在 §6.2.1 中对 RocksDB 的端到端评估捕捉了这些更实际、更高的上下文切换成本。

使用 KB_Timers 消除定时核心的优势。正如 §2 中所讨论的，为了分摊操作系统定时器的成本并避免争用 [55]，用户级运行时 [3, 30, 44, 54] 通常依赖于来自部分或完全专用于定时的核心的通知。在图 6 中，我们展示了使用两种不同操作系统定时器接口的定时核心的 CPU 利用率。在我们的实验中，我们使用 setitimer()，它在每个时间间隔传递一个信号，以及 nanosleep()，其中一个线程进入睡眠状态，然后在每个时间间隔被唤醒。UIPI 用于向其他核心发送通知，我们展示了随着需要通知的核心数量的增加，发送者（定时核心）的开销如何扩展。正如我们所见，操作系统定时器在较低的通知速率下会产生较小但仍然明显的开销。并且随着通知速率的增加，它们消耗的核心百分比越来越大。xUI 完全消除了这些成本，因为 KB_Timer 消除了使用昂贵的操作系统接口（每个定时器事件需要多次上下文切换）的需要，并且每个核心都可以有自己的定时器，从而消除了对 IPI 的需求。

更高性能的系统[30, 34, 44]有时会专门用一个完整的核心在高精度计时器（例如，rdtsc）上自旋，以避免操作系统开销。使用这种方法，我们发现我们可以在 5 微秒的抢占间隔下支持多达 22 个应用程序核心[30]。随着接收器核心数量的增加，通知成本 (senduipi) 变得占据主导地位。使用 KB_Timer，我们可以完全消除对专用计时器核心的需求，因此，作为一个基准，它为我们在 5 微秒时为每 22 个工作核心节省了 1 个核心。

当然，KB_Timer 的优势取决于可用核心的数量和工作负载。例如，在最理想的情况下，如果我们有两个核心，并且可以充分利用一个额外的核心来提高吞吐量，那么 KB_Timer 将使吞吐量翻倍。在最不理想的情况下，即22个其他工作核心被完全利用，它仅将吞吐量提高 4.5%。相对于 UIPI，KB_Timer 还提供了显著的单事件性能优势（图4）。

__最大中断延迟__。跟踪提供了最大化吞吐量的优势，因为它从不丢弃工作，但是，延迟取决于中断到达时正在执行的指令的性质。

为了探索这种限制，我们试图理解潜在的最坏情况延迟。为了测量这一点，我们构建了一个示例，该示例用一连串未命中 CPU 缓存的长延迟加载指令填充流水线——然后我们的依赖链最终生成堆栈指针寄存器的值。请注意，这是一个极端的病态情况，其中中断代码实际上依赖于一个很长的未完成指令链。虽然被中断的线程具有长链或修改堆栈指针并非不可能，但后者依赖于前者的可能性极低。

对于跟踪中断，在这种情况下，会延迟中断传递过程（§3.3）的执行，因为传递过程中有一条指令会读取堆栈指针寄存器的值（以将其保存在堆栈上）。该值在依赖链完成之前仍然不可用，可能会将中断传递延迟数百个周期。

当我们运行这个实验时，我们确实观察到最坏情况下的延迟接近 7000 个周期，其中包含 50 个或更多的长延迟加载链。利用英特尔的刷新机制，最坏情况下的延迟要小一个数量级（因为所有这些指令都被压缩了）。然而，我们使用更典型的基准测试（例如，linpack2、memops、fib）（以及我们的端到端工作负载）进行的实验证实，这确实是一个异常结果。在这些情况下，跟踪延迟明显优于刷新，这是因为在中断到达和通过跟踪发出之间通常没有延迟，而刷新会在中断发出之前引入额外的延迟。也就是说，我们认为这是一个我们正在对抗通过逆向工程和模拟可知性的极限的领域。因此，我们避免得出更明确的结论。

### 6.2 端到端评估

__6.2.1 Aspen中RocksDB的抢占__。抢占减少了请求服务时间差异显著的工作负载中的队头阻塞和尾部延迟[30, 37]，从而提高了有效吞吐量。我们使用三种配置评估了 Aspen 中 RocksDB 的这些优势（§5.3）：使用 UIPI SW Timer（UIPI + 专用定时器核心）和 xUI（KB_Timer + Tracking）的抢占，以及非抢占基线。同样，我们的工作负载是 99.5% 的 GET（1.2 μs）和 0.5% 的 SCAN（580 μs）请求的混合。我们选择 5 μs的抢占时间片，以优化 GET 吞吐量，并达到 1 ms 的尾部延迟目标。

图 7 展示了在不同负载下 GET 和 SCAN 请求的尾部延迟。在没有抢占的情况下，即使在非常低的负载下，GET 请求的尾部延迟也高达数十微秒。相比之下，通过 UIPIs 进行抢占，Aspen 能够在吞吐量超过每秒 100,000 个请求的情况下，将 GET 请求的尾部延迟保持在较低水平（最多几百微秒）。添加 KB Timers 和跟踪功能，由于接收 UIs 的开销较低，GET 请求的吞吐量进一步提高了 10%。在这两种情况下，抢占的代价是高负载下 SCAN 请求的尾部延迟略有升高。这是因为每个 SCAN 请求需要数百微秒才能完成，因此会被抢占多次。

__6.2.2 l3fwd 中的 IO 通知__。在此，我们比较轮询与 xUI（跟踪中断+中断转发）在将模拟高性能网卡上的设备中断传递到三层路由器（DPDK 的 l3fwd）时的性能和效率。xUI（跟踪中断+中断转发）在将模拟高性能网卡上的设备中断传递到三层路由器（DPDK 的 l3fwd）时的性能和效率。

我们观察到，两种方法的吞吐量几乎相同，xUI 的吞吐量比轮询低 0.08%。这是因为在使用设备 UI 时，中断处理程序在返回之前会再次轮询网络队列；在高负载下，处理程序永远不会返回，从而产生与轮询相同的吞吐量。对于 1、4 和 8 个网卡，xUI 实现的第 95 百分位尾部延迟分别在轮询延迟的 +2%、-8% 和 +65% 范围内。

我们还看到 xUI 如何降低 CPU 消耗，即留下更多的“空闲周期”，可用于其他有用的工作或节省电力。我们在图8中看到，在使用轮询时，所有 CPU 周期都用于处理数据包（“网络周期”）或轮询网络堆栈。相比之下，使用设备中断则不再需要轮询。因此，存在空闲周期。例如，在具有 1 个队列和 40% 负载的情况下，xUI 有 45% 的周期是空闲周期，而轮询总是消耗所有周期。

__6.2.3 DSA 中的 IO 通知__。在这里，我们探讨了 xUI 在传递来自我们模拟的、以 DSA 为模型的加速器的 I/O 完成事件方面的性能和效率优势。我们考虑两个标准：延迟——完成到达后多久被传递，以及空闲周期——在通知机制的成本之后，核心上还剩下多少周期。我们评估了接收 I/O 完成的三种选择：忙等待、定期轮询和 xUI 设备中断。

在此，我们看到了我们的三种策略及其结果：忙碌旋转（Busy spinning）—— 忙碌旋转持续轮询完成队列，这提供了极好的响应性（延迟），但浪费了所有未用于处理通知的 CPU 周期。

周期性轮询——定期检查完成情况——在我们的实验中，我们使用操作系统间隔计时器 (`setitimer()`) 实现这些检查。在这里我们看到，这释放了很大一部分周期，但也可能损害卸载任务的延迟，因为响应的到达时间变得不那么周期性/更嘈杂。例如，对于 20 微秒的请求，随着不可预测性的增加，周期性轮询的延迟会急剧增加。这是因为额外的响应延迟阻止了周期性轮询在请求完成时精确发生，导致处理必须等到下一个计时器事件。对于较短的请求，我们没有看到同样的效果，因为计时器频率已经非常高（2 𝜇s），几乎达到操作系统间隔计时器的极限，并且错过一个事件不会显着损害延迟。

相比之下，使用 xUI 时，延迟保持恒定，并且对于两种类型的请求，都在轮询所实现的延迟的 0.2 微秒范围内。总的来说，xUI 能够以最佳的 CPU 效率运行，仅有很小的延迟损失。例如，对于没有不可预测性（零附加噪声）的 2 微秒请求，跟踪中断可以释放 75% 的 CPU 周期。

## 7 相关工作

__被追踪的中断__。xUI 的被追踪中断通过减少未预测的控制流的影响来提高性能，这些控制流会削弱推测执行的优势。从这个意义上讲，它们类似于诸如谓词执行[50]或选择性刷新机制[20]等技术，这些技术试图在错误预测的分支方向与正确路径重新汇合时保留指令。

__用户中断的硬件支持__。先前的研究论证了用户级处理中断[51]和异常[57]的价值。据我们所知，英特尔于 2023 年发布的 Sapphire Rapids 处理器是首款为用户级中断提供硬件支持的商用高性能处理器。

RISC-V 规范的早期版本具有用户中断支持，但此扩展后来被删除 [24]。最近，RISC-V 添加了 CLIC 扩展 [16]，以支持嵌入式和实时应用的快速中断。虽然 xUI 建立在 UIPI 之上，但它的大部分内容适用于其他 ISA。

__加速器与致命的微秒__。Barroso 等人在他们的“Attack of the Killer Microseconds”[11]中指出，数据中心中越来越多的高性能 IO 设备（例如，网卡、闪存、非易失性存储器、加速器）将在微秒级的时间尺度上运行，而当前的处理器架构和软件无法在这种规模上提供事件通知、隐藏延迟等。他们进一步预测，加速器的日益普及将减少“数据中心税”，即常见的微秒级任务，如加密、压缩和序列化，这些任务消耗了谷歌数据中心 20-25% 的 CPU 周期[40]，只会加剧这个问题。

Intel 引入了 UIPI 以便高效地将 [38] 与一系列新的加速器 [63]（例如 DSA [42]）进行接口，这些加速器旨在解决数据中心的税收问题。xUI 直接解决了“杀手级微秒的攻击”中提出的一些挑战，并建立在 UIPI [38] 的愿景之上。

__用户层面的硬件虚拟化__。Dune [12] 利用了硬件虚拟化特性（VT [4] 和 EPT [4]）来提供用户层面对底层硬件特性的访问，包括中断系统。Shinjuku [37] 在此方法的基础上，在 UIPI 可用之前，实现了 IPI 的直接使用以进行抢占（而非信号）。然而，这种方法引入了额外的性能开销和部署挑战；因此，用户层面对虚拟化的访问从未得到广泛采用。

__高性能IO和轮询__。高性能用户空间 IO 栈几乎完全依赖轮询来检测 IO 事件 [13, 26, 37, 39, 41, 48, 49, 52, 62]。然而，轮询通常非常浪费资源，尤其是在流量不可预测以及队列数量增加时 [29, 43]。Hyperplane [47] 通过提供多队列 mwait 类型原语的专用加速器来减少轮询开销。通过 xUI，我们探索了如何完全消除轮询，证明中断可以在不牺牲效率的情况下实现低延迟和高性能。

__用户级抢占__。抢占是一种众所周知的技术，用于实现更精确的调度控制。通过抢占，调度器可以提高吞吐量，同时限制尾部延迟 [34, 37]。已经使用 UIPI 来实现这一点的系统 [25, 30, 35, 44] 可以从 xUI 提供的改进的性能和效率中受益。

在 Java 运行时开发者社区中，精确 GC 和高效抢占之间的张力早已为人所知[28]，并且 Go 开发者对此进行了广泛的探索[2, 10]。Java 虚拟线程中缺乏对循环抢占的支持[6]只是最新的一个例子。xUI 的硬件安全点为这个存在了几十年的问题提供了一个简单而有效的解决方案。

绕过 UIPI 限制。UIPI 官方不支持定时器中断。然而，Skyloft [35] 最近展示了一种绕过此限制的技巧，尽管方式非常受限。他们的方法首先设置 IA32_UINTR_MISC MSR 的 UINV 字段，该 MSR 由核心用于区分传统中断和 UIPI，以匹配本地 APIC 定时器的向量。因此，定时器中断被转发到当前线程。然而，本地 APIC 不会在定时器中断时设置本地核心上的 PIR（已发布中断）寄存器，这是传递 UIPI 所必需的。为了绕过这一点， Skyloft 以一种新颖的方式滥用了 senduipi。在启动时，它为所有线程设置 UPID 上的 SN 位。因此，当一个线程随后调用 senduipi 向自身发送 UIPI 时，PIR 将被设置，但不会传递 UIPI。因此，当定时器中断到达时，它将按预期传递。最后，Skyloft 在每次中断后的中断处理程序中重复此自发送 senduipi 技巧，以确保在下一次定时器中断之前设置 PIR。

不幸的是，这个技巧有很大的局限性。首先，因为只有一个本地 APIC 定时器，内核不再能使用本地 APIC 定时器进行调度、计时等。其次，这也禁用了用户中断的所有其他用途，首先是因为必须为 selfsenduipi 技巧设置 SN 位，更普遍的是因为为定时器中断重载 UINV 意味着处理器无法区分来自定时器中断的其他 UIPI。由于这些原因以及其他原因，例如直接的用户空间可编程性，这个技巧不能替代 xUI 的 KB_Timer 支持。

## 8 结论

xUI 为用户级别异步通知提供一流的支持。它在 UIPI 的性能和灵活性方面有了显著提高，并为共享内存轮询提供了一种更简单、更高效的替代方案。
